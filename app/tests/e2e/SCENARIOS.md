# E2E Scenario Catalog — Project Cockpit

Last updated: 2026-07-05 (Волна 4 — интеграция)

Постоянный контракт: каждый сценарий здесь = один `test()` в соответствующем spec-файле,
с тем же S-id в названии теста (по-русски).

**ВАЖНО (волна 3-4): приложение больше НЕ полностью read-only.** Home (список проектов)
теперь первый экран, а drafts/wizard/позиции нод персистятся в реальный
`window.localStorage` (см. `app/src/persistence/`: `storage.ts`, `projects.ts`,
`positions.ts`, и `cockpit.wizardDraft.v1` в `Wizard.tsx`). Карта Sales Bot по-прежнему
неизменяемый запечённый слепок и не редактируется напрямую — но пользователь теперь
может: создать/удалить черновик-проект, перетащить ноды (позиции персистятся),
автосохранить прогресс мастера. Тесты, которые создают черновики или двигают ноды,
ЧИСТЯТ `localStorage` в `beforeEach` через одноразовый `page.evaluate(() => localStorage.clear())`
(см. helpers/паттерн ниже) — не через `page.addInitScript`, см. ловушку в разделе
«Известные грабли» внизу файла.

Приложение: интерактивная карта проекта (React Flow) слоями L0→L4, запечённый слепок
Deadline Sales Bot, плюс домашний экран со списком проектов. Стек: Vite + React 19 + TS + Zustand.

**Общий хелпер** (`tests/e2e/helpers.ts`): `openSalesbot(page)` — `goto('/')` → дождаться
Home → клик по строке «Deadline Sales Bot» → дождаться канваса. `clickSalesbotRow(page)` —
то же самое, но без навигации (для сценариев после `page.reload()`, где `goto('/')` не нужен).
Все спеки, которым нужна карта, используют один из этих хелперов вместо повторения
логики входа вручную.

### S-LENS-06: Линза «Связи» — при hover нет дублей ребра одной пары (регрессия волны 0)
- **Given** карта внутри «Продукт / разработка» (L1, 17 рёбер — линза «Связи»)
- **When** снят набор `data-id` рёбер до и после наведения на `sys-instagram-module`
- **Then** оба набора не содержат повторов (`Set` того же размера, что массив), и hover не
  добавляет новых рёбер поверх уже нарисованных линзой (декорация на месте, не дублирование —
  см. фикс волны 0 `edgeId()` / волны 1 `FloatingEdge`)
- **Spec:** lenses.spec.ts
- **Destructive:** no

### S-LENS-07: Линза «Связи» — подписи-чипы рёбер видны без hover при ≤12 рёбрах уровня
- **Given** карта внутри Backend Core (L2, 10 нод / 6 рёбер — под порогом 12 из `FloatingEdge.showLabel`)
- **When** активна линза «Связи», курсор нигде не наведён
- **Then** хотя бы один `.react-flow__edgelabel-renderer` чип виден (в отличие от L1 org-product,
  где 17 рёбер > 12 и чипы скрыты до hover/select — не путать эти два уровня в новых тестах)
- **Spec:** lenses.spec.ts
- **Destructive:** no

- **When** `page.reload()`
- **Then** список проектов на Home идентичен (те же имена, тот же порядок) — черновик

- **When** клик по иконке корзины в строке черновика
- **Then** строка переключается в inline-подтверждение «Удалить «...»?» (НЕ `window.confirm()`);
  «Отмена» возвращает обычную строку без удаления; повторный клик «Удалить проект» → «Удалить»
  реально убирает черновик из списка и из `localStorage`
- **Spec:** home.spec.ts
- **Destructive:** no (удаляет только тестовый черновик, никогда не Sales Bot — см. S-HOME-05)
- **Примечание для тестов:** `ProjectRow` в состоянии подтверждения рендерит другой `<div>` без
  класса `.group` — не держать один и тот же `div.group`-локатор через обе фазы (поймали на
  первом прогоне волны 4), искать текст подтверждения через `page.getByText(...)` заново.

### S-HOME-05: Sales Bot неудаляем (нет кнопки удаления в его строке)

- **Then** в строке «Deadline Sales Bot» нет кнопки с aria-label «Удалить проект»

## Известные грабли (волна 4 — зафиксировано, чтобы не наступить снова)

1. **`page.addInitScript(() => window.localStorage.clear())` НЕЛЬЗЯ использовать в
   `beforeEach`, если тест внутри себя делает `page.reload()`.** Init-скрипты
   перевыполняются на КАЖДОЙ навигации фрейма, включая reload — поэтому localStorage
   стирается заново прямо перед проверкой персистентности, и любой reload-тест видит
   пустое хранилище независимо от того, сохранило ли приложение данные корректно.
   Симптом при отладке: `localStorage.getItem(...)` возвращает `null` сразу после
   `page.reload()`, хотя запись только что была подтверждена в той же странице.
   **Фикс:** одноразовая очистка — `await page.goto('/'); await page.evaluate(() =>
   window.localStorage.clear()); await page.goto('/')` (или `openSalesbot`/навигация
   вместо второго `goto`) в `beforeEach`, без `addInitScript`. См. `home.spec.ts`,
   `wizard.spec.ts`, `drag.spec.ts`.

2. **Не сравнивать позиции нод через `boundingBox()` (screen-space) между ДВУМЯ
   независимыми маунтами страницы** (напр. до/после `page.reload()`). `fitView`
   пересчитывает zoom/pan заново на каждом мануте и абсолютные экранные координаты не
   совпадают 1:1 даже при идентичной модельной позиции. Для персистентности сравнивайте
   сами данные (`localStorage.getItem('cockpit.positions.v1')`), а не пиксели. Внутри
   ОДНОГО маунта (без reload/перехода) `boundingBox()` сравнение валидно, но держите
   допуск ~3px, а не точный `toBeCloseTo(x, 0)` (0.5px) — суб-пиксельное дрожание от
   анимации `fitView` не считается регрессией.

3. **`header` содержит и кнопку-свитчер проекта, и L0-крошку с тем же текстом имени
   проекта** (волна 3 добавила постоянную L0-крошку). Простой `getByText('Имя проекта')`
   даёт `strict mode violation` (2 совпадения). Используйте
   `getByRole('button', { name: 'Имя проекта', exact: true })` — крошка имеет `L0 ` в
   accessible name и не матчит `exact: true`.

4. **`ProjectRow` (Home.tsx) рендерит РАЗНУЮ разметку в состоянии подтверждения
   удаления** — обычная строка обёрнута в `<div className="group relative">`,
   confirm-состояние — в `<div className="flex ...">` без класса `.group`. Не держите
   один и тот же `div.group`-локатор через обе фазы теста; после клика «Удалить проект»
   ищите текст подтверждения через `page.getByText(...)` заново.

5. **Кнопка «В корень» была удалена в волне 3** (коммит `ee6a575`) — заменена постоянной
   L0-крошкой в хлебных крошках. Старые тесты/скрипты, ссылающиеся на
   `getByRole('button', { name: 'В корень' })`, будут висеть 30s до таймаута, а не падать
   сразу — если видите такой таймаут, проверьте актуальность TopBar.tsx, а не считайте это
   флейком.
