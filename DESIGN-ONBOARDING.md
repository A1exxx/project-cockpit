## 0. ФАКТ-ЧЕКИ ПЕРЕД РАБОТОЙ (обязательно, до кода)

Судьи поймали три фактические ошибки в исходных предложениях — НЕ повторять:

### 4.2 Тонкая первая подсказка на канвасе (НЕ coach marks)
Вместо отклонённых плавающих coach-marks поверх нод (возня с координатами React Flow, плохой ROI по обоим судьям) — **одна статичная строка-хинт** внизу канваса при первом входе:
- Абсолютно позиционированная полоска у нижнего края области канваса (`z-10`, не перехватывает клики — `pointer-events-none`), `surface/90` + `border border-line`, `rounded-lg`, `font-mono text-[11px] text-ink-dim`, текст: `Клик по блоку — детали. Двойной клик — внутрь.`
- Показывается, пока `localStorage['cockpit.hintSeen'] !== '1'`. Гаснет (fade-out через opacity, 200ms) при первом `onNodeClick`/`onNodeDoubleClick` ИЛИ авто через ~8с; тогда ставит ключ `"1"`.
- Один раз за браузер. `prefers-reduced-motion` → без fade, просто снять.

Это учит ключевому скрытому жесту (двойной клик углубляет) в контексте, но без технической возни и без блокировки карты.

## 5. ЖИВОЙ КАНВАС — связи по наведению и выбору

Проблема (по обоим судьям): на линзах `blocks` и `risk` `project()` возвращает `edges:[]` (`projection.ts:250`) — холодный старт = 6 несвязанных коробок, хотя связей 64. Наведение и выбор на канвасе ничего не подсвечивают. Чиним так, чтобы «как связано» читалось без переключения в отдельную линзу.

### 5.1 Hover-раскрытие связей на линзах `blocks` и `risk`
- Подключить `onNodeMouseEnter`/`onNodeMouseLeave` в `Canvas.tsx` (сейчас не подключены; React Flow их поддерживает). Хранить `hoveredId` в локальном стейте канваса (НЕ в глобальном сторе — это чисто визуальное).
- При `hoveredId !== null` на линзах `blocks`/`risk`:
  - вычислить связи наведённого узла на видимом уровне через `rollupLinks(doc, visibleIds)` (готовое, `projection.ts:89`) — отфильтровать те, где `from===hoveredId || to===hoveredId`;
  - материализовать эти связи как временные рёбра (стиль `rolledToRfEdge`: `accent-dim`, dashed `6 4`, `strokeWidth 1.5`), `opacity` 0→0.6 (transition через CSS/motion на opacity);
  - соседние узлы (концы этих рёбер) — `opacity 1`;
  - все ОСТАЛЬНЫЕ узлы — `opacity 0.35` (dim). Реализовать через `data.dimmed` флаг, прокинутый в `CockpitNode`, который добавляет класс `opacity-35` (transition-opacity). НЕ анимировать цвет, только opacity.
  - `hoveredId → null`: всё возвращается (рёбра исчезают, dim снимается).
- Мемоизировать `rollupLinks` по видимому уровню один раз (в `useMemo` рядом с `projected`), фильтровать по hover дёшево. НЕ пересчитывать rollup в рендере на каждый mousemove.

### 5.2 Стойкая подсветка связей выбранного узла (все 4 линзы)
- Сейчас выбранный узел даёт только `ring-accent` (`CockpitNode.tsx:39`); связи не видны, а на blocks/risk их нет вовсе.
- Когда `selectedId !== null`: связи выбранного узла (тот же `rollupLinks`-фильтр по `selectedId`) показываются **стойко** (не только на hover) поверх текущей линзы — те же dashed accent-dim рёбра. Это синхронизирует канвас с блоком «Связи» в правой панели (`NodePanel LinksBlock`), который уже показывает их текстом.
- Приоритет: **select стойкий, hover временный поверх.** Если есть и selected, и hovered — показываем объединение, но dim применяем от hovered (hover — активное исследование). Чёткое правило, чтобы не мигало: `activeId = hoveredId ?? selectedId`; рёбра рисуем для `activeId`; dim остальных — только когда `hoveredId !== null` (при простом select без hover НЕ гасим остальные узлы, только рисуем рёбра выбранного + оставляем его `ring`).

### 5.3 Двусторонняя связь панель ↔ канвас (опционально, если бюджет)
Наведение на строку связи в `LinksBlock` (правая панель) → соответствующее ребро на канвасе кратко пульсирует opacity (один spring). Низкий приоритет — брать последним.

### 5.4 Motion-дисциплина
Всё строго `transform`/`opacity`, пружины DESIGN.md (stiffness 260 damping 26). Приглушение = opacity, НЕ цвет. Никакого glow. `prefers-reduced-motion` → появление/исчезание рёбер и dim = мгновенно/fade 100ms (глобальный клэмп покрывает CSS-переходы; для motion-компонентов уже стоит `MotionConfig reducedMotion="user"` в `App.tsx:27`).

## 8. SCOPE-ГРАНИЦА — что НЕ делаем (анти-перегруз)

Явный список отклонённого (reject обоих судей) — НЕ реализовывать:

1. **Модальный welcome-оверлей** поверх карты на старте. Нарушает «карта — герой»; юзер закроет не читая. Весь его контент живёт в НЕмодальном обзоре (§3) + кнопке тура (§4.1).
2. **Coach marks** (плавающие лейблы у нод/рейла с привязкой к координатам React Flow). Плохой ROI, возня с зумом, риск слопа. Замена — одна статичная строка-хинт (§4.2).
3. **Пульсация warn/risk-нод** («дышащие» бордеры) в линзе Риск. Ближе всего к «цирку» (MOTION_INTENSITY=6, «без цирка»); risk-статусов вообще 0. Внимание к зонам даёт кликабельный список (§3.2), не движение.
4. **Всегда-видимая полоса «О проекте» под шапкой** на корне. Дублирует обзор, ест вертикаль канваса. Тот же desc/stack уже в обзоре.
5. **`meta.plain` второй строкой на КАЖДОЙ ноде канваса.** Только в панели + tooltip (§7).
6. **Превью-ярлыки/стрелки направления на рёбрах, паттерны штриха call/data/event** — это углубление уже-рабочей линзы «Связи», вне scope холодного старта. Отложить.
7. Любые новые цвета/шрифты/эмодзи/glow/градиенты/карточки-плитки/графики-метрики. Никакого переписывания готового тура — только сделать находимым.
8. Не менять поведение мастера в этой волне (кроме того, что обзор/числа обязаны корректно работать на его картах через вычисление и fallback'и — это тестируется, но UI мастера не трогаем).
