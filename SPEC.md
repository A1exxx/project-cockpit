# Project Cockpit — полная спецификация системы
> Версия 0.2 · 2026-06-29 · статус: утверждение демо-scope
> Одна фраза: **система, в которой проект видно и создаётся слоями — как тело: от одежды до клеток.**

---

## 1. Видение

Инструмент для вайб-кодера (и для владельца проектов), который решает три боли:

1. **Понять чужой/свой большой проект** — открыть и увидеть устройство: слоями, с погружением до кода.
2. **Спроектировать новый проект** — система сама ведёт: задаёт вопросы, наставляет («сначала суть, потом визуал, потом фичи»), из ответов рождается карта будущего проекта.
3. **Управлять разработкой** — видеть, над какой частью работать, где риски/недоработки, и отдавать выбранный узел карты AI-агенту как готовую задачу.

Хранилище «понимания проекта»: карта не рисунок, а данные (JSON), поверх которых разные представления.

## 2. Метафора тела → уровни зума (ядро UX)

| Слой тела | Уровень | Что видно | Пример (Sales Bot) |
|---|---|---|---|
| Тело в одежде | **L0 Бизнес** | продукт + контекст: маркетинг, продажи, операции (серые блоки — «нас меньше касается, но видно, что к чему принадлежит») | Продукт · Маркетинг · Продажи |
| Кожа/силуэт | **L1 Системы** | крупные части: фронт, бэк, БД, интеграции, инфра | Backend · Admin UI · PostgreSQL · Telegram · Instagram · LLM |
| Кости/органы | **L2 Модули** | пакеты внутри системы | Bot Core · Instagram-модуль · Мозг · Лиды · Уведомления |
| Ткани | **L3 Фичи** | конкретные фичи/функции модуля + статус | Парсер комментов · DM-диалог · Approval-очередь |
| Клетки | **L4 Код** | реальный файл/функция: код, строки, кто вызывает | `approve_lead()` в `queue.py` |
| Кровеносные сосуды | **сквозной слой «Потоки»** | связи/данные текут через все уровни | лид: комментарий → DM → approval → CRM → уведомление |

Правила зума: клик/даблклик по узлу = погружение на уровень ниже; хлебные крошки сверху = путь («Sales Bot → Backend → Instagram → Approval-очередь»); Esc/клик по крошке = всплытие. Плавный переход (semantic zoom), не телепорт.

## 3. Линзы (виды «с разных сторон»)

Одни и те же данные — разные проекции:
1. **Блоки** (по умолчанию) — вложенные контейнеры-бенто: видно «что из чего состоит». Ближе всего к «телу».
2. **Связи** — граф: узлы текущего уровня + сквозные рёбра (call/data/event), подсветка «кровеносной системы».
3. **Риск** — тепловая карта: красный (недоработки/hotspots), жёлтый (внимание), зелёный (стабильно), серый (todo — задумано, не сделано).
4. **Дерево** — классическая иерархия для тех, кому так проще.
5. **Поток** (после демо) — путь сущности через систему по шагам.

## 4. AI-гид (система «ведёт, направляет, наставляет»)

Панель-компаньон (правая колонка). Три режима:

- **Наставник (создание проекта).** Мастер-интервью по логике PROJECT-OS: 1) суть — проблема/один пользователь/один исход; 2) визуал — какие экраны/точки входа; 3) фичи — must / «НЕ сейчас»; 4) данные — какие сущности храним; 5) связи — интеграции/потоки; 6) план v1 — Done-чеклист + дата-убийца. Каждый шаг = вопросы гида → ответы пользователя → на карте вырастают узлы (status=todo). Результат: карта-черновик + чартер.
- **Аналитик (понимание).** Вопросы к карте: «где самые рискованные места», «что заденет изменение Мозга», «что не доделано». Ответы со ссылками на узлы (клик = переход).
- **Диспетчер (делегирование).** Выбран узел → «Сформировать задачу для агента»: markdown-промпт с контекстом узла (путь, символы, связи, статус, критерии приёмки) → копировать / отправить в Claude Code. Карта = источник задач.

**AI-провайдер:** Gemini 2.5 Flash (ключ есть — `GOOGLE_API_KEY` в `D:\Христианский бот\.env`; API поддерживает вызов из браузера). Режимы работы гида: (а) локально — ключ из `.env.local`; (б) публичная демо — BYO-ключ (localStorage, в репо не попадает) + **заскриптованный сценарий** как fallback, чтобы демо кликалась у любого без ключа. Позже — свой тонкий бэкенд-прокси (v3+).

## 5. Данные (сердце системы)

`project-map.json` — единый формат карты:
```json
{
  "version": 1,
  "project": {"name": "…", "desc": "…", "stack": ["…"]},
  "nodes": [{
    "id": "kebab", "parent": "id|null", "level": 0,
    "kind": "org|system|module|feature|code",
    "title": "…", "sub": "…",
    "status": "ok|warn|risk|todo",
    "meta": {"path": "…", "symbol": "…", "lines": 0, "note": "…"},
    "code": "только для L4, ≤60 строк"
  }],
  "links": [{"from": "id", "to": "id", "kind": "call|data|event", "label": "…"}]
}
```
Источники карты:
- **Реальный код** → экспортёр из jCodeMunch (outline, dependency graph, hotspots, dead code) → nodes/links. В демо — запечённый слепок; в v2+ — кнопка «обновить с репо».
- **Замысел** → мастер-интервью гида → узлы status=todo.
- **Слияние** (киллер-фича, v4): план ↔ реальность на одной карте; todo без кода = недоработка, код без плана = неучтённое.

## 6. Демо-версия (текущая цель) — scope

**Что демонстрируем:** «открыл → понял проект слоями → нашёл риски → погрузился до кода → спросил гида → сгенерил задачу агенту → (отдельный поток) создал карту нового проекта через интервью».

Done-чеклист демо:
- [ ] Полноэкранное приложение (не виджет): канвас + крошки + линзы + панель узла + панель гида.
- [ ] Реальные данные: слепок **Deadline Sales Bot** (read-only, запечённый JSON).
- [ ] Зум L0→L4: бизнес-блоки → системы → модули → фичи → реальный код с подсветкой.
- [ ] Линзы: Блоки · Связи · Риск · Дерево.
- [ ] Панель узла: статус, связи (кликабельные), недоработки, для L4 — код.
- [ ] Гид: заскриптованный сценарий из коробки + живой Gemini при вставленном ключе; кнопка «Задача для агента» на узле.
- [ ] Мастер «Новый проект» (6 шагов) → карта-черновик рядом с Sales Bot (демо-поток, можно с предзаполненным примером).
- [ ] Read-only: никакого редактирования карты в демо.
- [ ] Задеплоено: GitHub repo (A1exxx) + GitHub Pages, открывается с телефона.

**НЕ в демо:** редактирование карты, живой реиндекс репо, мультипроект-дашборд, аккаунты, биллинг, бэкенд, слияние план↔реальность (показываем идею одним статичным экраном-тизером, не механикой).

## 7. Архитектура демо

- **Стек:** Vite + React + TypeScript + React Flow (канвас, semantic zoom, custom nodes) + Zustand (состояние) + Tailwind. Код: `D:\project-cockpit\app`.
- **Слои кода:** `data/` (типы + salesbot-map.json + wizard-шаблоны) · `graph/` (map→React Flow проекции: по линзе и уровню) · `ui/` (канвас, крошки, линзы, панели) · `guide/` (сценарий + Gemini-клиент) · `export/` (генератор «задачи для агента»).
- **Экспортёр слепка:** отдельный скрипт-процесс (Claude + jCodeMunch → JSON), в демо результат запечён.
- **Деплой:** GitHub Pages (`gh-pages` ветка), как на других проектах A1exxx.

## 8. Дорога после демо
- **v2 Живые данные:** экспортёр по кнопке для любого локального репо (jCodeMunch), обновление карты, несколько проектов.
- **v3 Живой AI везде:** свой прокси-бэкенд (ключ не у пользователя), аналитик на реальных данных jCodeMunch, диспетчер шлёт задачи в Claude Code напрямую.
- **v4 План↔Реальность + продукт:** слияние карт, diff-подсветка недоработок, мультипроект, free-tier для вайб-кодеров, монетизация.

## 9. Риски проекта (честно)
- **Сложность = ловушка.** Демо уже большая; всё, что не в Done-чеклисте, — в «НЕ сейчас», иначе не дошиппим (см. PROJECT-OS, kill date 2026-07-13).
- **Красивый зум — самая дорогая часть.** Если semantic zoom в React Flow начнёт съедать дни — упрощаем до «уровни = переходы с анимацией», механика важнее спецэффекта.
- **Gemini в статике** — ключ в публичный билд не кладём никогда; только BYO/скрипт.
